# 第一部分：真缺陷（3 條，約 1.5 小時）

每項先寫一個**修補前會紅**的測試，再修。

## 1. `PUT /api/admin/settings` 邊驗證邊寫入 → 部分套用

**`src/routes/api/admin/settings.ts:147-260`** — **全稽核唯一無爭議的實害 bug。**

讀完整個函式後，範圍比初判更大。這個模式貫穿全部十幾個設定鍵：

```ts
if ("brand" in body)      { ... await put("brand", brand); }        // 立即寫入
if ("contact_url" in body){ ... return json({error:"bad-input"},400); }  // ← 後面才可能失敗
for (const k of QUOTA_KEYS)   { ... await put(k, String(n)); ... return json(...,400); }
if ("relay_meter" in body)    { ... await put/del }
if ("demo_mode" in body)      { ... }
for (const k of DEMO_NUM_KEYS){ ... await put(k, ...); ... return json(...,400); }
if ("tg_bot_token" in body)   { ... }
```

**任何一個鍵驗證失敗，它前面所有已寫入的鍵都留在 D1，且沒有任何 rollback。**
前端（`src/routes/settings.ts:373-380`）把 400 當整包失敗、不重抓狀態
→ **UI 顯示的值跟資料庫不一致，而且使用者不會知道。**

沒有任何測試或註解把這當成預期行為：`test/int/settings.test.ts:73` 有測
`quota_pg_day: 0 → 400`，但**只送單一鍵**，這條路徑從沒被觸及。

**修**：全部驗證 → 收集 statements → 一次 `env.DB.batch()`。
副作用是把十幾次 D1 round-trip 收成 1 次。
**測**：多鍵請求 ＋ 後段鍵不合法 → 斷言前段鍵**未**變動。

## 2. 上游錯誤內文可能含第三方金鑰

**`src/routes/api/playground/chat.ts:181-194`** → **`src/lib/observe.ts:25`**

已確認 `observe.ts` **沒有任何既存遮罩**：

```ts
const detail = String((extra && extra.detail) || (e && e.stack) || "").slice(0, 2000);
await env.DB.prepare("INSERT INTO errlog (ts,src,msg,detail,user_id,path) VALUES ...")
```

而 `chat.ts:181` 把上游回應原文塞進 `detail`。上游 401/403 內文常回顯金鑰——
OpenAI 有遮罩，但這個 gateway 主打的 `kind: "custom"` 廉價 OpenAI 相容轉售商經常回顯完整金鑰。
落進 `errlog.detail` → `/api/admin/errors` 讀得到 → 進 R2 備份（`cron.ts:105`）。

**修**：在 `observe.ts` 的 `reportErrorNow` 統一遮罩
`sk-[A-Za-z0-9_-]{16,}` / `AIza[\w-]{30,}` / `uak-\w+`。
**放在 observe 層而非呼叫點**，未來新增的 `reportError` 自動受保護。
成本趨近於零，沒有反面論證。

## 3. `visits.path` 沒截斷

**`src/routes/_middleware.ts:52`**

```ts
url.pathname + (url.search || ""),          // ← 沒有 .slice()
(h.get("user-agent") || "").slice(0, 700),
(h.get("accept-language") || "").slice(0, 200),
(h.get("referer") || "").slice(0, 500),
```

**隔壁每一欄都截斷了，只有 `path` 沒有**，且 `migrations/0001_baseline.sql:8` 的
`visits.path` 是無長度限制的 TEXT。`visitLog` 對每個請求都跑（`src/router.ts:107`），
`shouldLog`（`_middleware.ts:27-32`）只要帶 `Accept: text/html` 就對**任意路徑**回 true。
CF 接受約 16 KB URL。

這個不對稱很難說是有意的。**修**：`.slice(0, 500)`，一行。

> **保留期不放這裡。** 初版把「`visits`／`errlog` 無保留策略」列為緊急，過度了——
> 目前流量約 123 次瀏覽/天，一年約 45k 列，離問題還很遠。轉 DEBT（見第 7 項）。
> 另外初版建議清 `audit_log` 是**錯的**，稽核紀錄本來就該長期保留。

## 7. THREAT-MODEL 與 DEBT 更新

**`docs/THREAT-MODEL.md` 新增「已知接受風險」一節：**

- **`goodOrigin` 對正式網址放行 localhost Origin**（`auth.ts:237`，`test/unit/auth.test.ts:88-91` 記錄意圖）。
  風險是真的：localhost 來源的頁面可對正式站發帶 cookie 的 simple-request POST
  （preflight 擋 PUT/DELETE，但 `content-type: text/plain` 帶 JSON body 的 POST 不觸發 preflight）。
  利用需要受害管理員載入攻擊者控制的 localhost 頁面——很窄。
  **寫清楚：什麼換什麼、什麼情況下會改變決定。不改程式碼。**
- **修正 §2.4**：「path 片段重新編碼」防止 traversal 的宣稱不精確——
  `encodeURIComponent` 不轉義 `.`（`relay/[[path]].ts:83-86`）。host 不可控、僅在已授權上游內。

**`DEBT.md` 新增（各附還債門檻）：**

| 項目 | 門檻 |
|---|---|
| `visits`／`errlog` 無保留策略 | 任一表破 50 萬列 |
| `backupToR2` 整庫累積成單一記憶體字串（`cron.ts:119-139`，isolate 上限 128 MB） | 單次備份輸出破 20 MB |

# 第四部分：真的讓履歷更有力

每一項通過同一個測試：**面試時它能不能變成一句你講得出來、對方會追問的話。**

## 8. `docs/AUDIT-2026-07.md` — 本版最有辨識度的產出

一句話版本：

> 「我對自己的正式系統做了結構化安全稽核。找到 **3 個真缺陷**，
> **並把 4 個初判為漏洞的項目重新歸類為非缺陷，每一個都附推翻依據。**」

**第二個分句才是重點。** 找 bug 很多人會；**分辨 bug 與取捨**才是 review 的難處，
也正好補上這個專案「技術難度」的缺口——批判性 review 是實打實的難技能。

**必須寫進去的四個推翻（不要藏，它們才是這份文件的價值）：**

1. **`goodOrigin` localhost** — 初判 CSRF 漏洞。查 `test/unit/auth.test.ts:88-91` 發現
   測試名稱就叫「localhost 開發放行」→ 改判已接受風險。
   **教訓：讀原始碼不讀測試會製造偽陽性。**
2. **前端未進工具鏈** — 初判可信度落差。查 `ADR-0008:29` 發現早已明文揭露 → 改判已揭露取捨。
3. **前端「四個 bug」** — 初判 4 個缺陷。逐行讀完發現 `isNaN(Date)` 靠隱式轉型正確、
   `opt.year="numeric"` 在 JS 合法、`f[1](r)` 正常運作 → **四個都是型別推論的抱怨，程式碼是對的**。
   **教訓：靜態檢查工具的警告不等於缺陷。**
4. **`cc.cd` cookie tossing** — 初判可被同後綴網域寫入 session cookie。
   **實際比對 Public Suffix List 確認 `cc.cd` 在清單上** → 不成立。
   「我去查了而不是假設」比多列一條假漏洞更有說服力。

**另附一節「測試套件自我檢查」：**

> `test/int/router.test.ts:113-116` 的測試名稱是「管理員 API 帶金鑰 → **200**」，
> 但斷言是 `expect([200, 401]).toContain(r.status)`——**授權成功或失敗都會通過**，
> 而這是 `api/admin/users/index.ts` 在 vitest 裡唯一的接觸點。原註解顯示作者知道 setup 不對，
> 選擇放寬斷言而非修正 setup。

（初版說「兩條」是錯的——`:101` 的 `expect([404, 502])` 實際上有效：
它斷言請求已通過金鑰驗證抵達渠道查詢，401／403 會讓它失敗。只有 `:113` 是真的失效。）

「我在自己 321 條測試裡找出一條測了等於沒測的」是很強的一句話，修它 20 分鐘。

**格式**：每筆＝發現／可利用性評估／`file:line` 證據／**判定（缺陷 or 取捨 or 誤判）**／
修補 commit 或推翻依據。

## 9. DO 限流器的併發壓測 → `docs/REPORT.md`

**目標很窄，不要做成 benchmark 套件。一個腳本、一個可辯護的主張。**

`docs/REPORT.md` 現在最弱的是 `n=10`（誠實標註了，但仍然弱）。而這個專案**最難的部分是
DO 限流器**，目前證據是 `test/unit/rate-limiter.test.ts:86-92` 的 in-process
`Promise.all` 30 發、恰好 10 過——好測試，但不是**真 HTTP 併發**。

新增 `tools/loadtest.mjs`，沿用**已存在**的 `tools/mock-upstream.mjs` ＋ `wrangler dev`：

1. **打 200 個真並行 HTTP，斷言恰好 `limit` 個 200、其餘 429**
2. **Gateway overhead 分佈**：mock 上游延遲可控，`(total − upstream)` 就是這個 worker
   自己的成本。n≈2000 的 p50/p95/p99

`docs/REPORT.md` 加一節，**明確標示是本機合成壓測、不是正式流量**——
現有那份的誠實語氣是它的優點，不要為了數字好看動它。

**為什麼值得做**：讓「我實作了原子限流器」變成
**「我實作了原子限流器，並在 200 個並行 HTTP 請求下驗證恰好放行 N 個」**。
後者是面試官會追問的句子，前者不是。

## 10. ADR 防禦練習（**最高投報率，零程式碼**）

git log 顯示 11 天、64 commit、15,700 行程式碼 ＋ 2,464 行文件。
**面試官看得到這個時間線，而且一定會問。** 問題不會是「你是不是用了 AI」——2026 年那不是問題——
會是這種形式：

1. 為什麼 demo 是 **fail-closed**、會員配額是 **fail-open**？同一個 DO、相反政策，為什麼？
2. 限流器為什麼是原子的？改用 D1 `COUNT` 會怎樣？（`rate-limiter.test.ts:86-92` 就是答案）
3. CSP 為什麼同時用 per-request nonce（SSR）跟 sha256（靜態）？
4. **ADR-0005 為什麼用 pump 不用 `tee()`**？（已確認 ADR 原文：`tee()` 是「顯而易見的實作」，
   你否決了它——講得出理由這題就贏）
5. ADR-0003 選共享金鑰＋配額而不做 BYOK，什麼情況下你會改變決定？
6. `goodOrigin` 為什麼放行 localhost？你知道它的風險嗎？（**第 7 項的文件就是這題的答案**）

**產出**：`docs/DEFENSE.md`（gitignore 或放本機）——每題三到五句白話，不看筆記能講。

**這件事的投報率高於你接下來寫的任何一行程式碼。** 這個專案的價值是高變異的：
守得住 → 履歷最強項；卡住一題 → 落差會被看見，比交一個更小但完全自己想的專案還糟。
