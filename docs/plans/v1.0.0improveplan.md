# ipua v1.0.0 品質升級計畫

## Context（為什麼做）

uaip.cc.cd 目前功能完整（CMS、會員、LLM 中轉、VPN、Playground）但工程證據為零：無測試、無 CI、無配額、無錯誤日誌、無安全文件。目標是把專案從「興趣等級」升到「履歷可講的等級」——不加新服務，而是補齊**可驗證（測試）、可量測（計量與延遲數據）、可重現（一鍵本機環境）、可論述（ADR／threat model／競品比較）**四塊，最後部署並打 `v1.0.0` tag。

### 使用者已拍板的決定
1. 中轉維持**共享金鑰**＋計量配額（BYOK 留 v2）
2. 告警**先只做站內**（D1 errlog＋管理頁；無 Telegram/Sentry）
3. 文件**雙語**（英文為主＋繁中）
4. CI **只跑測試**，部署照舊本機 wrangler
5. **VPN 隱藏**：非站長且未批准 vpn 服務者完全看不到（選單不渲染、/vpn 裝成不存在、/api/me 不吐 vpn 欄位）
6. 全自動做完 → 部署 → remote migration → tag v1.0.0 push

### 關鍵架構事實（探索已驗證）
- SSR 殼 `pageShell()`（lib/site.js:119-152）**不讀 session**，選單 `menuHtml()` 對所有人相同；會員 UI 靠前端打 /api/me
- 三個服務頁全是「永遠 200 的殼＋前端閘門」；唯一真閘門在資料端點（relay 401/403、vpn/sub 403）
- SSR 回應標頭只在 `html()`（site.js:73-78）設定 → CSP 插入點
- `lib/apidoc.js` 由 `node tools/build-apidoc.mjs` 從 API.md 生成，**不可手改**
- 無 package.json / node_modules / .github —— 工具鏈是綠地
- 現有計量僅 `relay_calls`/`vpn_pulls` 計數器；**無任何限流**；錯誤全被 `catch(e){}` 吞掉
- LOGS_TOKEN 明文寫在 ADMIN.md:76 且已 commit（私有 repo，但要輪替＋搬走）
- 未知路徑走 SPA fallback 回 200 index.html → /vpn 要「隱形」需模擬此行為（`env.ASSETS.fetch`）

### 規劃階段的重要設計裁決
- **中轉計量不用 `tee()`**：客戶端中斷時 tee 的另一支會繼續讀，上游持續生成＝燒錢。改用 chat.js 已驗證的 pump 模式（讀一次、寫給客戶端、順手掃 usage；寫入失敗→cancel 上游）。加 settings `relay_meter='0'` 保險開關可即時退回純直通。
- **站長完全豁免配額**（與全站「站長全通」一致，避免自己的 agent 被 429）
- Playground 配額檢查**必須在任何 D1 寫入之前**（否則 429 還是會建對話）
- CSP：SSR 用 per-request **nonce**（在 `html()` 單一入口 regex 蓋章到所有 `<script`）；靜態 SPA 用 **sha256 hash** 寫進 `_headers`＋CI 防漂移檢查。全站審計過**零** inline `on*=` 屬性，script 不需 `'unsafe-inline'`；style 保留 unsafe-inline（記入 DEBT）
- `db/schema.sql` 退役，migrations/ 成唯一 schema 來源（測試用 `readD1Migrations` 套用，避免雙軌漂移）
- relay 的 model 從**回應**掃出（回應 JSON 帶 "model"），絕不緩衝會員請求 body
- `stream_options.include_usage` 只加給 `kind==='openai'`（custom 的本地服務可能拒收未知欄位）
